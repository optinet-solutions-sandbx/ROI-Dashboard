Migration Approaches

Three ways to convert your Vite/React app to Next.js + Tailwind. Pick the one that fits.

A

Direct Migration

Keep the same tab-based navigation (state-driven, single route at /). Convert CSS → Tailwind. Add /api/upload route for server-side Excel parsing. Minimal restructuring — same component tree, same logic.

Pros

  • Fastest to ship
  • Lowest risk
  • Same mental model

Cons

  • No URL-based routing
  • Can't bookmark a page
  • Not fully idiomatic Next.js
B

URL-Based Routing

Each section becomes its own route: /overview, /affiliates, /campaigns, etc. Data shared via React Context provider. Add /api/upload route. Navigation uses Next.js <Link>.

Pros

  • Idiomatic Next.js
  • Bookmarkable URLs
  • Better scalability

Cons

  • Context needed for data sharing
  • More restructuring
  • Slightly more complex
C

Full Next.js Patterns

URL routing + loading.tsx / error.tsx per route + React Server Components where possible (layout, sidebar). Client components only where interactivity is needed. Most "Next.js-native" architecture.

Pros

  • Best performance
  • Most Next.js-native
  • Proper loading/error states

Cons

  • Most restructuring
  • Charts/recharts are client-only
  • Highest complexity

My recommendation

Option B — URL-based routing gives you a proper Next.js app without the complexity of full server components. Since all your charts (Recharts) and interactivity are client-side anyway, Option C's server component benefits are minimal. Option A is fine but wastes Next.js's routing strengths.
